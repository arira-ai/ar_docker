# Demo - Entrypoint vs CMD

This is easily one of the most notoriously confusing topics in Docker. Both `CMD` and `ENTRYPOINT` do exactly the same basic thing: they tell the container what command to run when it starts.

The difference lies entirely in **how they behave when a user tries to pass arguments at the end of a `docker run` command.**

Before we look at the specific scenarios, use this interactive simulator to see how `ENTRYPOINT`, `CMD`, and runtime arguments combine to form the final command.

Here is the breakdown of the exact scenarios for when to use which instruction, along with a code demo.

---

### The Golden Rule

Always use the **Exec Form** (JSON array: `["executable", "param1"]`) rather than the **Shell Form** (`executable param1`). 

>If you use the Shell form, Docker wraps your command in `/bin/sh -c`, which ignores signals (like `CTRL+C`) and breaks the ability of `CMD` and `ENTRYPOINT` to work together.

---

### Scenario 1: When to opt for `CMD` only

**Use case:** 

General-purpose environments, background services, or applications where you want to give the user the freedom to easily override the default behavior and run an entirely different command (like a bash shell).

**How it works:** 

Whatever you put in `CMD` is the default. If the user types anything after the image name in `docker run`, the `CMD` is completely ignored and replaced.

**Demo:**

```dockerfile
# Dockerfile
FROM ubuntu:latest
CMD ["echo", "Hello from the default CMD"]

```

**Outcome:**

* `docker run my-ubuntu` -> Outputs: *"Hello from the default CMD"*
* `docker run my-ubuntu python3` -> The `echo` is wiped out. It runs Python instead.
* `docker run -it my-ubuntu /bin/bash` -> Wipes out the `echo` and drops you into a terminal.

---

### Scenario 2: When to opt for `ENTRYPOINT` only

**Use case:** 

When your container is built to do **one specific thing** and act like a dedicated executable/CLI tool. You want to force the container to run this command, and any arguments the user passes should just be fed into that command.

**How it works:** 

The `ENTRYPOINT` cannot be easily overridden (a user has to explicitly use the `--entrypoint` flag). Anything typed at the end of `docker run` is appended to the `ENTRYPOINT` as parameters.

**Demo:**

```dockerfile
# Dockerfile
FROM alpine:latest
ENTRYPOINT ["ping"]

```

**Outcome:**

* `docker run my-ping` -> Outputs an error because `ping` is missing a destination.
* `docker run my-ping google.com` -> Translates to `ping google.com`.
* `docker run my-ping -c 3 localhost` -> Translates to `ping -c 3 localhost`.
* `docker run my-ping /bin/bash` -> Translates to `ping /bin/bash` (which fails, proving the user is locked into the `ping` executable).

---

### Scenario 3: When to use BOTH (The Best Practice)

**Use case:** 

You are building a CLI tool or a dedicated application, but you want it to work out-of-the-box without the user needing to type extra arguments, while still allowing them to override the *target* of the tool.

**How it works:** 

`ENTRYPOINT` sets the hardcoded executable. `CMD` provides the default arguments for that executable. If the user passes runtime arguments, they replace `CMD` but keep `ENTRYPOINT`.

**Demo:**

```dockerfile
# Dockerfile
FROM alpine:latest
ENTRYPOINT ["ping"]
CMD ["localhost"]

```

**Outcome:**

* `docker run my-smart-ping` -> Translates to `ping localhost` (Uses `ENTRYPOINT` + `CMD`).
* `docker run my-smart-ping google.com` -> Translates to `ping google.com` (`google.com` overrides `localhost`, but `ping` stays).

### Summary Cheat Sheet

| Setup in Dockerfile | You run: `docker run my-image` | You run: `docker run my-image arg1` |
| --- | --- | --- |
| `CMD ["echo", "hi"]` | Runs: `echo hi` | Runs: `arg1` (replaces CMD) |
| `ENTRYPOINT ["echo"]` | Runs: `echo` | Runs: `echo arg1` (appends) |
| `ENTRYPOINT ["echo"]` + `CMD ["hi"]` | Runs: `echo hi` | Runs: `echo arg1` (replaces CMD, appends to ENTRYPOINT) |
